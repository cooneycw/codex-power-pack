# Codex skills

Most skill dirs in this directory are **generated**, not authored here. They are
pulled from [claude-power-pack](https://github.com/cooneycw/claude-power-pack),
which authors the shared command families once in `.claude/commands/<family>/` and
generates the Codex `SKILL.md` surface with `scripts/codex-skill-sync.py`. This repo
consumes that generated output (the "pull" side of the single-source-of-truth
bridge, codex-power-pack#75) and carries narrow CxPP-owned runtime-state
adaptations where Codex must use `.codex/` paths instead of Claude-owned
`.claude/` paths.

Each `<family>-<command>/` dir carries a `<!-- GENERATED by claude-power-pack ... -->`
marker. `claude-md` is **not** pulled (Out-of-Scope for CxPP; the Codex-native
`agents-md` family covers the AGENTS.md world).

## CxPP-owned native skills

The `agents-md-*` skill dirs are authored in this repo because they govern
Codex's `AGENTS.md` instruction surface. The `project-next` and `project-lite`
skill dirs are also authored here because CPP advertises those project commands
but does not generate Codex skill payloads for them. These native skills
intentionally do not carry the generated marker and are excluded from
`vendor/claude-power-pack/codex-skills.sha256`.

Keep this native surface narrow. Add new CxPP-owned skill dirs only when the
family is genuinely Codex-specific, then update `scripts/codex_skills_sync.py`
and tests so the ownership boundary stays explicit.

## Do not hand-edit generated files

A hand-edit is caught by the CI drift gate (`make codex-skills-check`,
`scripts/codex_skills_sync.py --check`), which fails when any file here diverges from
the manifest recorded in `vendor/claude-power-pack/`. For shared command behavior,
edit upstream in CPP and refresh. For CxPP-owned runtime-state adaptations such as
`.codex/worktrees/` and `.codex/friction.jsonl`, update the generated skill
surface here, keep plugin payloads byte-identical, add/adjust tests, and
re-snapshot the manifest.

## Harness lint

Generated skills also run through `make harness-lint`
(`scripts/harness_lint.py --check`) so Claude-only constructs do not reach Codex
unadapted. A literal `AskUserQuestion`, `/plugin`, `CLAUDE.md`, `Agent tool`,
`Skill tool`, or leading `!` command is allowed only when the skill's `SKILL.md`
contains a matching Codex harness adaptation. A literal `.claude/worktrees` path
is stricter: it is rejected unless the line explicitly marks itself as CPP or
Claude source context. Codex runtime worktrees belong under `.codex/worktrees/`.

If a literal token must remain for documentation, add a reviewed exception to
`.codex/harness-lint-allowlist.txt` using `rule|path|needle|reason`. Prefer
rewriting the skill or adding an adaptation over allowlisting.

## Reconcile rule

To change shared generated skill behavior, edit the **source** upstream, not the
copy here:

1. Edit `.claude/commands/<family>/<command>.md` in claude-power-pack.
2. Regenerate there: `make codex-skills`.
3. Re-pull here: `make codex-skills-refresh CPP_ROOT=<path> CPP_REF=<sha>`
   (updates `vendor/claude-power-pack/PIN` and the drift manifest).

The pinned upstream commit lives in `vendor/claude-power-pack/PIN`.

To change a CxPP-owned runtime-state adaptation, patch the generated skill copy in
this repo, mirror it into the matching plugin payload, and run
`scripts/codex_skills_sync.py --write` plus the local verification gates.

To change a CxPP-owned native skill such as `agents-md-lint` or `project-next`,
edit its skill package here and run the local verification gates.
